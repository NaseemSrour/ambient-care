# Deployment notes (what we did, and why)

A plain-language record of how Ambient Care got online, for future-me. The
formal how-to is in [DEPLOYMENT.md](DEPLOYMENT.md); this is the "what actually
happened + why" version.

## The goal
Host the backend (FastAPI) so the TRMNL device and the family phones can reach
it 24/7 over HTTPS — **for free**, and **always-on** (no cold-start delays).

---

## Step 1 — Render.com (the hosting)

**What Render is:** a platform that takes our GitHub repo, builds the
`Dockerfile`, and runs it on a public HTTPS URL. It reads `render.yaml` in the
repo to configure everything automatically ("Blueprint" deploy).

**What I did:**
1. Signed in to render.com with GitHub.
2. **New + → Blueprint → picked the `ambient-care` repo.**
3. During this, **I installed the "Render" GitHub App** into my GitHub account
   and granted it access to the repo. ⚠️ *Remember this:* that authorization
   lives in **GitHub → Settings → Applications → Installed GitHub Apps → Render**.
   If I ever revoke it there, Render loses access and auto-deploys stop.
4. Set the env var **`AC_FAMILY_PASSCODE`** (the family login code).
   `AC_JWT_SECRET` and `AC_TRMNL_POLL_KEY` were auto-generated by Render.
5. Applied → Render built the Docker image (Flutter web build + Python) and
   went live.

**Why free plan:** this is a tiny personal app; the free plan is plenty.

**The catch with free:** a Render free service **spins down after ~15 minutes
of no traffic**, and the next request has to wait for a cold start. That would
make the device's poll occasionally slow/fail. → solved in Step 3.

**No persistent disk on free:** the SQLite database resets if the instance is
redeployed or recycled. We decided that's fine here (messages expire in 24h
anyway; tasks/verses are quick to re-add).

**Auto-deploy:** because it's a Blueprint from GitHub, **pushing to `main`
triggers a new deploy automatically.**

---

## Step 2 — Verify it's live

Checked in the browser:
- `https://<my-service>.onrender.com/api/health` → `{"status":"ok"}`
- `/` → the family dashboard (logged in with the passcode)
- `/api/trmnl.html?key=<poll key>` → the elder's screen rendered visually.

---

## Step 3 — UptimeRobot (keep it awake)

**What UptimeRobot is:** a free website monitor that pings a URL on a schedule.

**What I did:** created a free account, **Add New Monitor → HTTP(s)**, URL set
to `https://<my-service>.onrender.com/api/health`, interval **5 minutes**.

**Why:** a request every 5 minutes is more frequent than Render's 15-minute
idle timeout, so the service **never sleeps** → effectively always-on, still
$0. (~730 hours/month, within Render's free 750-hour allowance.)

---

## About `AC_TRMNL_POLL_KEY` (the secret in the device URL)

**What it is:** a shared secret that protects the two device-facing endpoints
(`/api/trmnl-payload` and `/api/trmnl.html`). The elder's screen contains
personal info (messages, visits), so these endpoints require `?key=<secret>`
to match — otherwise they return `{"detail":"forbidden"}`.

**How it got its value:** `render.yaml` has `generateValue: true` for it, so
**Render generated a random value automatically at deploy time.**

**Where to find it:** Render → the service → **Environment** tab → reveal
`AC_TRMNL_POLL_KEY` → copy.

**Gotcha we hit:** the generated value contains symbols (like `+ / =`) that
break inside a URL. **The fix was to URL-encode the key** before putting it
after `?key=`. (Alternative: set it manually to a letters/numbers-only value.)

**Expected behavior:**
- correct key → the endpoint returns the screen (HTML) / payload (JSON).
- wrong or missing key → `{"detail":"forbidden"}`.

The device will use this same key in its polling URL (next step: TRMNL plugin).

## Updating the app later (redeploy)

Render auto-deploys on every push to `main`, so shipping a change is just:

```powershell
git add -A
git commit -m "describe the change"
git push origin main
```

Then Render automatically:
1. Detects the push to `main`.
2. **Rebuilds the whole Docker image** — this rebuilds *both* the Flutter web app
   and the Python backend, so I never commit `frontend/build/` (built in the cloud).
3. Redeploys (a few minutes; watch Render → service → Events/Logs; live when
   `/api/health` passes).
4. The **device shows the change on its next poll** (up to the 30-min refresh),
   or force-refresh from the TRMNL app.

Remember:
- Backend-only and frontend changes use the **same** command — just push.
- **No local `flutter build` needed** for Render; that's only for previewing
  single-origin mode on my own machine.
- ⚠️ A redeploy **resets the database** (free plan, no disk): custom tasks,
  verses, and events return to seeded defaults and must be re-added in the app.
  Messages don't matter (they expire in 24h).

## Quick reference

| Thing | Where |
|---|---|
| Hosting | render.com → service **ambient-care** |
| Env vars / secrets | Render → service → **Environment** |

| Render's GitHub access | GitHub → Settings → Installed GitHub Apps → Render |
| Redeploy | just `git push origin main` |
